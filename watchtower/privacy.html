Watchtower · Privacy Policy

Watchtower is an internal monitoring tool Umbrella Automation runs on the Windows endpoints we manage for our clients. This page describes what the agent reports back, where that data is stored, who can read it, and how long we keep it.

Last updated: 2026-05-24

Watchtower is operational telemetry for an existing managed-services relationship. It is not a consumer product. We only install it on endpoints owned by our MSP clients under a signed service agreement, and the data it collects only goes to Umbrella Automation staff — not sold, not shared with third parties for marketing, not used for any purpose outside of supporting and securing your IT environment.

1. What we collect

The agent runs once per day on each monitored endpoint and reports a single check-in payload to our Cloudflare Worker, which writes it to our Firestore database. The categories below are the complete list of fields the agent can populate — nothing else is collected, recorded, or transmitted.

Identity & system

CategoryWhat it contains
Host identityComputer name, workgroup or AD domain, anonymous per-install UUID (pcId), the client name bound to the install token, IP addresses (external + per-NIC internal).
HardwareManufacturer, model, BIOS version + date, service tag/serial, CPU model + core count, total RAM + per-DIMM details, TPM presence + spec version, virtualization platform (Hyper-V / VMware / KVM / etc.) and on Hyper-V guests the parent host's name.
Operating systemWindows edition, version, build, install date, last boot time, architecture (x64/ARM).
StorageVolume letters, filesystem type, size + free space, fixed/removable status.
NetworkNIC names, MACs, IPv4 addresses, link speed, default-gateway flag.
User accountsLocal user account names, enabled/disabled state, last logon timestamp, password-expiry settings. No passwords or password hashes.

Software inventory

CategoryWhat it contains
Installed applicationsNames, versions, publishers from the Windows Uninstall registry. Used to see what's on each machine for support and license-tracking purposes.
Windows hotfixesKB numbers + install dates for installed Windows patches.
Pending Windows UpdatesCounts + titles of updates pending install (queried from the Windows Update Agent), plus reboot-pending state.
USB device historyFriendly name + serial number of USB devices that have been connected to the machine (from the registry's persistent device-arrival log).

Backup & security state

CategoryWhat it contains
Backup productsVeeam (Backup & Replication + Agent), Windows Server Backup, Carbonite: install state, version, last backup result, last successful backup date, recent job history, target destinations, retention policy.
Microsoft DefenderEnabled state, real-time protection state, definition version + age, last scan timestamps.
SentinelOneInstall state + agent version + service state. No telemetry from the SentinelOne agent itself — just whether it's installed and running.
Dell OMSA / iDRACOn Dell hardware: RAID controller info, virtual disk (array) health, physical disk health with vendor + serial + SMART predictive-failure flag.
LogMeInInstall state, version, service state, Central computer description (the friendly host label set by the technician).

What we explicitly do NOT collect

  • File contents of any kind. We see which applications are installed but not the documents or data those applications hold.
  • Browser history, bookmarks, saved passwords, cookies, autofill.
  • Email content, instant messages, or any communication payloads.
  • Keystrokes, screen recordings, webcam input, or microphone audio.
  • Personal files in user profiles (Documents, Pictures, Downloads, etc.).
  • The contents of any database or LOB application running on the host.
  • Passwords, password hashes, encryption keys, or authentication tokens for any service.
  • Active Directory directory data beyond the local computer's domain membership.

2. Why we collect it

Solely to operate, support, and secure the IT environments we manage. Specific use cases:

  • Health monitoring. Spot a failing RAID disk, a backup that hasn't run in 3 days, a server with 90+ days uptime that needs reboot, or a host whose external IP just changed unexpectedly.
  • Patch awareness. Know which endpoints have critical Windows Updates pending so we can schedule maintenance windows.
  • Asset inventory. Maintain an accurate per-client list of hardware models, OS versions, installed software, and license counts for support + renewal planning.
  • Incident response. When something breaks, we need to know what version of which product is on which machine without dialing in.
  • Change detection. Alert on new endpoints joining the fleet, hosts going dark, or USB devices being plugged in.

We do not use the data for advertising, profiling, sale, model training, or any purpose other than the ones above.

4. Who can read it

  • Umbrella Automation staff with verified @umbrellaautomation.com Google accounts. The dashboard enforces this gate via Firebase Authentication + email-domain check on every read. There are no other admin roles.
  • Cloudflare and Google infrastructure staff to the extent their own service obligations require (e.g. infrastructure maintenance). Both vendors are GDPR + SOC 2 audited.
  • We do not share Watchtower data with any other Umbrella Automation customer. Per-client tokens enforce that data from your endpoints stays scoped to your client record.

5. Retention

  • Current per-host status: kept indefinitely while the endpoint is active. Replaced on each daily check-in.
  • Per-host history (timeline of check-ins, IP changes, events): kept for the lifetime of the endpoint in our fleet.
  • Decommissioned endpoints: the record is preserved (marked decommissioned) until the client requests its deletion or we close the account.
  • Activity log: append-only; preserved for audit purposes.
  • Email notifications: held by Resend for 90 days then aged out.

If your service agreement with us ends, we delete all your Watchtower records within 90 days of account closure. You may request deletion sooner; contact info below.

6. Security

  • Every agent-to-worker request authenticates with a per-client install token (43 bytes of base64url entropy; SHA-256 hashed server-side — the raw token is shown to the operator once at generation, never stored).
  • All transport is TLS 1.2+ to *.workers.dev, *.googleapis.com, and api.resend.com.
  • Firestore writes are scoped to a service account; raw firewall is closed via declarative rules that only the worker (via service account) and verified MSP-admin browsers can read.
  • Dashboard sign-in requires Google's verified @umbrellaautomation.com account — no shared passwords, no API keys in the browser.
  • The agent runs as LocalSystem on the endpoint. It does not open inbound network ports; all communication is agent-initiated, outbound HTTPS only.

7. Your control

If you're an Umbrella Automation client whose endpoints are monitored:

  • Uninstall the agent at any time. Add/Remove Programs → "Umbrella Watchtower" → Uninstall. The agent phones home one last time to mark the endpoint decommissioned in our dashboard, then removes itself.
  • Request a copy of what we have on a specific endpoint. Email us (address below) with the hostname and we'll export the full record.
  • Request deletion. Same path: email with the hostname or client name. Records are removed within 30 days of confirmation.
  • Audit what we've changed recently. The Activity log inside the dashboard records every admin action (host reassign, client rename, decommission, force-update, token generate / revoke). We can share an excerpt scoped to your endpoints on request.
