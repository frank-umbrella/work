Watchtower Operator's Guide

Everything you need to roll out the Watchtower agent on a client PC, customize the install, and tear it down cleanly.

What Watchtower is

A small Windows agent that calls home once a day with the host's external IP, asset inventory, backup status, RAID/disk health, and the state of the management/security software you typically rely on. The dashboard shows the fleet and lets you toggle per-PC kill switches.

Per-host probes:

  • System · network · storage · users · software · hotfixes · USB history — Belarc-lite asset inventory pulled from WMI + registry.
  • Defender — enabled, realtime status, definitions version + age, last scan times.
  • Veeam — detects Backup & Replication and/or Agent for Windows side-by-side (both can be installed on the same host), with last-session result.
  • Windows Server Backup — Get-WBSummary on Server SKUs. Last backup time, last successful backup, retained versions, next scheduled backup, current operation.
  • Carbonite — presence + version + service state for Server Backup, Endpoint Backup, or Safe variants.
  • LogMeIn — install state, service + guardian state, Central computer description.
  • SentinelOne — agent version + service state.
  • Dell OMSA — on hosts with OMSA installed, pulls controllers, virtual disks (RAID arrays), and physical disks via omreport. Surfaces a worst-case rollup so a single failed drive shows up as a red badge in the dashboard at a glance.
  • iDRAC Service Module — on Dell servers with iSM installed, reports version + service state. (BMC firmware version is a future addition.)

Two processes on each client PC:

  • watchtower-svc.exe — runs as the WatchtowerAgent Windows service under LocalSystem. Does the probes + check-in.
  • watchtower-tray.exe — runs in the interactive user session, shows a status icon in the system tray.

Quick start

The 60-second path. You build Watchtower-Setup.exe once (or grab the most recent build from the dist folder) and ship the same EXE to every client — the token is entered at install time.

  1. Open the dashboard → Clients tab → + New client (if needed) → click + Token on the client's row. Copy the token (you only get to see it once).
  2. Run Watchtower-Setup.exe as admin on the target PC. Paste the token into the wizard when prompted; the installer pings Watchtower to verify the token before continuing.
  3. Wait ~60 seconds for the first check-in. The host appears in the dashboard's Fleet tab.
Why only one installer? Older versions baked the token into the EXE at build time, so you needed a custom build per client. Since v0.3.0 the installer carries no per-client data — the worker resolves the client name from the token at install time and the wizard writes the right config.

Install

Build the installer (once)

You only need to do this when the agent code itself changes — not per client. Prerequisites on your build box:

  • Windows 10/11 with admin rights
  • Python 3.11+ — winget install Python.Python.3.11
  • PyInstaller and the agent's Python deps — pip install pyinstaller pywin32 pystray pillow requests
  • Inno Setup 6 — winget install JRSoftware.InnoSetup -e

Then:

cd work\watchtower\installer
.\build.ps1

Output lands at installer\dist\Watchtower-Setup.exe. Keep that EXE around — it's the same file you'll deploy to every client.

Deploy to a client PC

  1. Generate a token in the dashboard. Clients tab → (client) → + Token. Copy it immediately — Watchtower only stores the SHA-256 hash, so it can't show you the raw value again.
  2. Get the EXE to the target PC. Push Watchtower-Setup.exe however you normally distribute software (LogMeIn, RMM, copy-paste, email, file share — it's a self-contained EXE).
  3. Run it as admin. Double-click works (UAC will prompt). For unattended deploys see silent install below.
  4. Paste the token at the wizard prompt. The installer makes a quick call to the Watchtower worker to verify the token. If the token is wrong, revoked, or unreachable the wizard refuses to advance — nothing is written to the PC. On success it proceeds and writes %ProgramData%\Watchtower\config.json with the token + worker-resolved client name + a fresh per-install pcId.
Network requirement at install time: the wizard needs to reach the worker to validate the token. If the target PC is behind a strict firewall, allow outbound HTTPS to watchtower-worker.umbrelladev.workers.dev before running the installer.

Verify check-in

After install, the service kicks off its first check-in within a minute. Then:

  • Dashboard → Fleet: the host appears at the top, status Healthy.
  • If it doesn't show up, on the target PC run sc.exe query WatchtowerAgent — should report STATE: 4 RUNNING.
  • Force a re-check-in: sc.exe stop WatchtowerAgent; sc.exe start WatchtowerAgent

Customize install

Silent / RMM install

For unattended deploys, pass the token on the command line — the wizard is skipped entirely:

Watchtower-Setup.exe /VERYSILENT /SUPPRESSMSGBOXES /NORESTART /TOKEN=wt_xxxxx

Optional overrides:

FlagPurpose
/TOKEN=<wt_...>Install token. Required for silent installs (no wizard to enter it). Validated against the worker before install completes — bad token = non-zero exit code.
/CLIENT="Acme Corp"Override the client label shown in Add/Remove Programs and the tray tooltip. If omitted, the worker's token-bound client name is used.
/WORKERURL=https://...Hit a different worker (staging / testing only). Defaults to production.
/COMPONENTS="logmein"Only relevant when the installer was built with -LogmeinMsi. Default behavior is to include LogMeIn install; pass /COMPONENTS="" to explicitly skip it.
/VERYSILENTStandard Inno Setup — suppresses the wizard.
/SUPPRESSMSGBOXESSuppress error dialogs (failures still produce non-zero exit codes and write to the install log).

Exit code 0 = success. Non-zero = something failed; check the install log at %TEMP%\Setup Log YYYY-MM-DD #NNN.txt.

build.ps1 parameters

Build-time customization (rare — the defaults are right for production):

ParameterDefaultNotes
-WorkerUrlhttps://watchtower-worker.umbrelladev.workers.devBaked-in default the wizard pings. Override only when you need installs to default to a staging worker.
-AppVersion0.3.0Cosmetic — appears in Add/Remove Programs and the wizard title.
-LogmeinMsi(none)Optional path to a LogMeIn host MSI. When set, the wizard shows an opt-out checkbox for installing LogMeIn alongside the agent — see below.
-LogmeinMsiArgs(none)Optional extra args to append to the LogMeIn msiexec call. Used when the MSI isn't pre-customized (e.g. needs DEPLOYID=...). /quiet /norestart are always passed by the .iss regardless.
-SkipPyInstalleroffReuse cached build\*.exe when iterating on the .iss. Saves ~2 minutes per build.
No -ClientName or -InstallToken at build time. Both are runtime inputs as of v0.3.0 — either prompted in the wizard or passed via /TOKEN= /CLIENT= for silent installs.

Bundle LogMeIn MSI

Optional. If you have a LogMeIn host MSI you typically install alongside the agent, you can bake it into the installer EXE so a single deploy puts both on the target. The wizard shows a checkbox Also install LogMeIn remote access (checked by default) on a Components page after token validation; operators can uncheck it on hosts that already have LogMeIn or don't need it.

If your MSI is a "Deploy Installation Package" download from LogMeIn Central

That's the common case — the DEPLOYID is baked into the MSI itself, so you don't need to pass any LogMeIn properties on the command line. Just point -LogmeinMsi at the file:

cd work\watchtower\installer
.\build.ps1 -LogmeinMsi "D:\Downloads\LogMeIn.msi"

If your MSI is the generic LogMeIn host MSI

You'll need to pass DEPLOYID + a couple of related properties so the host registers with your LogMeIn Central account. Per the LogMeIn Central deployment docs:

.\build.ps1 `
  -LogmeinMsi "D:\Downloads\LogMeIn.msi" `
  -LogmeinMsiArgs "DEPLOYID=xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx INSTALLMETHOD=5 FQDNDESC=1"

Common LogMeIn MSI properties you might want:

PropertyPurpose
DEPLOYIDYour deployment package ID from LogMeIn Central. Find it under Deploy Installation Package in the Central console.
INSTALLMETHOD=5Standard install method.
FQDNDESC=1Display fully qualified domain name as the host name in Central.
LMIDESCRIPTION="..."Set the computer name shown in Central. Otherwise the FQDN/hostname is used.
LMIACCESSCODE=...Set an access code (password) for connecting to this host.

How the LogMeIn install runs

  • Wizard install — user sees a Components page with the LogMeIn checkbox checked by default; can uncheck per install.
  • Silent install with default args — LogMeIn installs too.
  • Silent install skipping LogMeIn — Watchtower-Setup.exe /VERYSILENT /TOKEN=wt_xxx /COMPONENTS=""
  • MSI is invoked as msiexec /i /quiet /norestart <your -LogmeinMsiArgs here> — no UI flashes, no reboot triggered.
The MSI is bundled, not downloaded. The installer never reaches out for it at runtime — the MSI is extracted from inside the EXE to %TEMP%, installed, then deleted. Works on offline target machines as long as the MSI itself is offline-installable (most LogMeIn MSIs are).
The installer\vendor\ directory is gitignored. Drop the MSI there (or anywhere else on your dev box); it won't accidentally get committed.

When -LogmeinMsi isn't passed, the resulting installer has no Components page at all and behaves exactly like a non-LogMeIn build.

Custom install directory

Defaults to C:\Program Files\Umbrella Watchtower\. Override with:

Watchtower-Setup.exe /VERYSILENT /TOKEN=wt_xxxxx /DIR="D:\Tools\Watchtower"
Note: %ProgramData%\Watchtower\ (for config.json and state.json) is always used regardless of /DIR. That path is shared between the service and the per-user tray.

Alternate worker URL

When the install token belongs to a different worker (staging, a parallel test environment, etc.), pass it on the command line so both the wizard's validation call AND the resulting config.json use it:

Watchtower-Setup.exe /TOKEN=wt_xxxxx /WORKERURL=https://watchtower-worker-staging.umbrelladev.workers.dev

Uninstall

Remote (preferred)

From the dashboard:

  1. Fleet tab → click the row for the host you want to remove.
  2. In the drawer, click Decommission.
  3. The agent self-uninstalls on its next check-in (within 24h). The dashboard row stays until you manually delete it — useful for audit.
Heads up: Decommission flips uninstall: true on the per-agent config doc. If the PC is offline at decommission time it will pick up the flag whenever it next reaches the worker. If it never reaches the worker again, the install stays in place — you'll need to uninstall locally.

Local (UI or CLI)

UI: Settings → Apps → Installed apps → Umbrella Watchtower Agent → Uninstall.

CLI (elevated PowerShell, in case the GUI uninstaller is broken or unreachable):

sc.exe stop WatchtowerAgent
sc.exe delete WatchtowerAgent
taskkill /im watchtower-tray.exe /f
Remove-Item "C:\Program Files\Umbrella Watchtower" -Recurse -Force
Remove-Item "C:\ProgramData\Watchtower" -Recurse -Force
Remove-ItemProperty -Path "HKLM:\SOFTWARE\Microsoft\Windows\CurrentVersion\Run" -Name "WatchtowerTray" -ErrorAction SilentlyContinue

Install tokens

Each client gets its own install token. The dashboard generates 32 cryptographically-random bytes (256 bits of entropy), base64url-encodes them, and prefixes wt_ for easy spotting in logs. Only the SHA-256 hash is stored in Firestore — you see the raw token exactly once, in the modal that pops up right after generation.

Paste the token into the wizard at install time (or pass /TOKEN=... for silent installs). The installer validates it against the worker before completing and writes the verified token to %ProgramData%\Watchtower\config.json on the target PC. Don't share tokens across clients — the worker enforces the binding on every check-in, and using a token outside its assigned client makes alert routing wrong.

Revoking a token

From the Clients tab, click Tokens on the client's row to expand the token list, then click Revoke on the token in question. The token doc is kept (with revoked: true) so you have an audit trail; the worker treats revoked + unknown identically so an attacker can't tell which tokens ever existed.

Agents using a revoked token will start returning 401 on check-in. They keep running locally — they just stop reporting until you push a rebuilt installer with a fresh token.

Dell-aware UX

When a host's manufacturer reports as Dell, the dashboard layers in two extras automatically. No agent change needed — the manufacturer field is already in the system probe.

Service Tag + Express Service Code

The Service Tag (from Win32_BIOS.SerialNumber — Dell's BIOS reports it there) becomes a clickable link straight to that host's Dell Support page:

https://www.dell.com/support/home/en-us/product-support/servicetag/<tag>

Next to the tag we show the Express Service Code — the same value converted from base-36 to base-10 (e.g. 5BXYZ12 → 12345650770). Some Dell support phone trees only accept the numeric code, so having it pre-computed saves a step. Non-Dell hosts keep showing the serial as plain text (no link, no Express code).

What gets captured

Per-probe summary

Each daily check-in posts a payload to the worker with a report object containing one key per probe that found anything. Probes silently skip themselves when the product isn't installed (e.g. omsa doesn't appear on an HP host).

All probes are auto-detect. Nothing to configure per host or per client. Drop the agent on a server that already has Veeam, Carbonite, LogMeIn, SentinelOne, OMSA, iDRAC iSM, Windows Server Backup, etc. and each section populates on the next check-in (within ~60 seconds of install). If a product isn't installed, its section is omitted from the report entirely — the drawer just doesn't show it for that host.
FieldWhat's in it
systemHostname, domain/workgroup, manufacturer, model, Service Tag, BIOS version + date, full OS info (name, build, install date, last boot, edition, architecture), CPU (name + cores + speed + sockets), memory (total + per-DIMM), TPM (present, enabled, spec version).
networkPer-NIC: name, MAC, IPv4 addresses, link speed. External IP (resolved via the worker's cf-connecting-ip header on check-in).
storagePer-volume: drive letter, filesystem, type (fixed/removable), total + free GB.
usersLocal user accounts with status (enabled/disabled, password-required, lockout), plus recent interactive logon history.
softwareEvery installed app from the Uninstall registry tree: name, version, publisher. Count + full list.
defenderEnabled, realtime status, definitions version + age, last quick + full scan timestamps, AV engine + product versions.
veeamPer Veeam product (B&R and/or Agent for Windows): edition, version, last session result + start/end times.
wsbWindows Server Backup (server SKUs only): last backup time, last successful backup time (these diverge on failure), next scheduled, retained versions, current operation, last result mapped from HRESULT, plus a recentJobs array carrying the last 10 backup attempts (Get-WBJob -Previous 10) with per-job result + error description.
carboniteProduct flavor + version + publisher (Server Backup / Endpoint / Safe), service state.
logmeinInstall state, agent version, service state, guardian-service state, Central computer description.
sentineloneAgent version, service state.
omsaDell OpenManage: per-controller info (model, firmware, driver, status), virtual disks (RAID arrays with layout + size + state), physical disks (vendor, product, capacity, status, SMART predictive-failure flag). Worst-case rollup so a single bad disk shows red.
idracDell iDRAC Service Module: version, service state.
hotfixesTotal count + the most recent ~10 (ID + install date).
usbUSB device history from the registry: friendly name, product, serial. (Last-used timestamps are a roadmap item.)
probeErrorsAny probes that threw during collection — recorded but don't block the rest of the report.

Intake email on first check-in

The first time a brand-new host's agent reports in (no existing record in the dashboard), the worker fires a one-time intake email summarizing what the probes found on that machine. Think of it as a "welcome to the fleet, here's what's on this box" report you can archive for your records.

When it fires

  • Once per host. Triggered inside the /checkin handler when there's no previous /agents/{pcId} doc in Firestore (the firstSeen flag in history). Subsequent check-ins from the same host don't re-fire it.
  • Per-PC kill switches apply. The same config.enabled and config.emailEnabled flags that gate the IP-change and backup-failure emails also gate this one. (Defaults are true, so a fresh agent that's never had its config touched WILL get an intake email.)
  • Honored across reinstalls — the installer preserves pcId when re-running over an existing install, so a re-install of the same agent on the same machine does NOT generate a second intake email.

What's in it

A formatted HTML summary with:

  • Identity — hostname, client, external IP, pcId, agent version
  • Hardware — manufacturer, model, Service Tag (with Dell support link if Dell), CPU, RAM, TPM, BIOS
  • Operating system — Windows version, build, install date, domain/workgroup
  • Volumes — per-drive size + free space
  • Network interfaces — all NICs with IPv4 addresses
  • Storage — Dell OMSA RAID rollup + disk/array counts (only if OMSA detected)
  • Backups — Veeam products + last-job result, WSB status, Carbonite products (whichever are present)
  • Security — Defender status + definitions, SentinelOne (if present)
  • Remote access — LogMeIn details, iDRAC iSM (if present)
  • Inventory — installed app count, hotfix count, USB device count

Sections for products that aren't installed on this host are skipped, not shown empty.

Backup monitoring

Windows Server Backup gets first-class treatment because backup failures are the kind of thing you want to know about immediately, not the next time someone happens to look at the dashboard.

Fleet-level visibility

When a host's most recent backup attempt didn't succeed, an extra badge appears next to its Status in the Fleet table:

  • Backup failing — latest attempt failed, but a successful backup is less than 3 days old.
  • 5d no backup — 3 or more days since the last successful backup. The number is the actual count.
  • No backup on record — WSB is installed but has never completed a successful backup.

The Status filter dropdown has a "Backup failing" option that narrows the fleet to just the hosts with any of the above states.

Per-host drawer

Opening a host's drawer shows a red banner at the top of the Windows Server Backup section when the latest attempt failed, with the gap to last successful backup and the WSB-reported error message. Below the standard summary, a Recent backup attempts details disclosure expands the last 10 jobs with their individual results and errors — useful for spotting "started failing 3 days ago" versus "this is a one-day blip".

Email alerts on new failures

The worker fires a Resend email when a check-in arrives showing a new failed backup attempt. Deduping is based on the WSB lastBackupTime field: one email per new failed attempt, not one per check-in for an ongoing failure. So a daily backup that fails 5 days in a row produces 5 emails (one per day), not dozens.

  • Honors the per-PC kill switches — flipping emailEnabled off in the drawer silences backup alerts for that host the same way it silences IP-change alerts.
  • Webhook payload — if webhookEnabled is on, a JSON POST also fires with event: "wsb_backup_failed" and the full failure context (result, attemptedAt, lastSuccess, daysSinceSuccess).
  • Recovery — when a backup eventually succeeds, no email fires (no point celebrating a normal day). The next failure starts the cycle again.

Troubleshooting

Agent installed but not appearing in dashboard

  • Confirm the service is running: sc.exe query WatchtowerAgent — STATE: 4 RUNNING.
  • Check the worker logs in the Cloudflare dashboard for the most recent check-in. 401s mean a bad token (typo at install time, or revoked since). 5xx means worker or Firestore is having a bad day. Note: with the v0.3.0 wizard, 401s on first check-in should be rare — the installer pre-validates against the worker.
  • Worker URL mismatch: open %ProgramData%\Watchtower\config.json on the target PC and confirm workerUrl matches what's deployed.

Service won't start

  • Event Viewer → Windows Logs → System → filter on Source = Service Control Manager. Most failures are a missing dependency in the PyInstaller bundle — rebuild without -SkipPyInstaller.
  • Run the EXE directly to see the error: "C:\Program Files\Umbrella Watchtower\watchtower-svc.exe" from an elevated prompt.

Tray icon missing on RDP'd servers

The tray is registered in HKLM\…\Run, which fires once per interactive logon. RDP into a server creates a new interactive session — the tray should launch on the second login. If it doesn't, check Task Manager for watchtower-tray.exe; if it's missing, run it manually from the install dir.

"Token revoked" by accident

You can't un-revoke (intentional — revocation is a one-way audit event). Generate a fresh token, rebuild the installer, and re-deploy. The pcId is preserved across reinstalls so the host stays the same dashboard row.
